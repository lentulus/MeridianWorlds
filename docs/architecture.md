# Worlds Pipeline — Architecture

*2026-05-07*

---

## Overview

Three concerns, three stores, no overlap:

```
world.db (SQLite)       canonical reference data — species, worlds, settlement
Meridian API            physical body data — fetched at generation time, never cached locally
Obsidian vault          narrative prose — generated by Claude, keyed by obsidian_slug
Python enums            GURPS generation tables — grammar, not data; never persisted
```

The pipeline direction is always left-to-right. `world.db` is the source of
truth for everything the worlds project owns. Meridian is read-only at the
boundary. The vault is write-only output.

---

## 1. world.db (SQLite)

Path defined in `config.py` as `WORLDS_DB`. Apply schema with:

```
sqlite3 ~/databases/world.db < sql/create_species.sql
```

### Tables

| Table | Rows | Purpose |
|---|---|---|
| `world_settlement` | one per surveyed body | geologic activity, RVM, habitability, settlement data |
| `species` | one per named species | full GURPS Chapter 6 trait set |
| `species_traits` | many per species | GURPS racial advantages/disadvantages |

See `docs/species_schema.md` for full DDL.

### Design decisions

**SQLite over Parquet.** Species and settlement counts are in the dozens-to-hundreds
range. Parquet's columnar compression and distributed scan offer no benefit at
this scale, and introduce a second tool dependency (PyArrow/DuckDB) purely for
overhead. SQLite handles the JOIN between `species` and `species_traits` natively
and is universally queryable.

**One database, not two.** An earlier plan split canonical data (world.db) from
campaign-mutable data (story.db). The split is deferred: for a single campaign
the cross-database JOIN cost is not worth it. Campaign-time mutations are recorded
in a `system_events` table within `world.db` when that need arises.

**world.db, not starfield.db.** `starfield.db` is not a dependency. Physical body
data is owned by the Meridian project and accessed through its API. If an
offline cache is ever needed, it lives here as a separate table — not as a
direct file path.

## 2. Meridian API boundary (`meridian/`)

Meridian owns star systems and physical body data. The worlds project does not
import Meridian internals; it calls functions in `meridian/api.py` and receives
typed Pydantic objects.

```
meridian/
  __init__.py
  models.py     — Body, System (Pydantic BaseModel)
  api.py        — get_body(), get_system(), list_bodies() — stubs until API is live
```

### Contract

Meridian returns JSON dicts. Translation to typed objects happens once, in
`meridian/api.py`, at the boundary. The rest of the codebase uses `Body` and
`System` objects with attribute access and IDE support.

```python
# meridian/api.py — what each function must eventually do:
def get_body(body_id: str) -> Body:
    raw: dict = meridian.call("get_body", body_id)   # JSON in
    return Body.model_validate(raw)                  # typed out

def get_system(system_id: str) -> System: ...
def list_bodies(system_id: str) -> list[Body]: ...
```

### Scope

The boundary layer covers **any data Meridian owns**, not only physical bodies.
Ship position is the next expected addition — dynamic world-state that Meridian
owns and the worlds project consumes read-only. When that work begins, extend
`meridian/models.py` with a `ShipPosition` model and add the stub to
`meridian/api.py`. The pattern is identical to `Body`.

### Body fields (SI units — see §5)

| Field | Type | Unit |
|---|---|---|
| `body_id` | str | opaque key |
| `name` | str | — |
| `system_id` | str | opaque key |
| `gravity_g` | float | standard gravities (1 g = 9.807 m s⁻²) |
| `atm_pressure_bar` | float \| None | bar (1 bar = 100 kPa) |
| `mean_temp_k` | float \| None | kelvin |
| `hydrosphere_pct` | float \| None | % surface coverage |
| `volatile_type` | str \| None | dominant volatile (e.g. WATER, AMMONIA) |

`body_id` is an opaque key obtained from Meridian. `world_settlement.body_id`
and `species.body_id` store it as a plain TEXT column; foreign-key integrity is
enforced at application level, not by SQLite, because the backing store is
external.

## 3. Generation grammar (`src/generators/` — not yet created)

GURPS Space Tables I–IX (pp. 136–170) are probability distributions over traits.
They are encoded as Python `StrEnum` classes and weighted-choice dicts, **not**
as database rows. Location when built: `src/generators/alien_enums.py` and
`src/generators/alien_tables.py`.

The entry point is a pure function:

```python
def roll_species(rng: random.Random, body: Body | None) -> dict:
    """Return a fully-populated species row dict, ready for world.db insert."""
```

No database access inside the generator. The caller inserts the result.

## 4. Obsidian vault output

```
~/projects/reference/worldbuild/
  species/
    <obsidian_slug>.md
```

Each file contains YAML frontmatter (all mechanical fields from `species`) plus
narrative prose sections written by Claude. The frontmatter is the authoritative
copy for Obsidian/Dataview queries. The `species` row in `world.db` is the
authoritative copy for the pipeline.

The `obsidian_slug` column in `species` is the bridge. Naming convention:
`{name_lower_snake_case}`, with a UUID suffix on collision.

### Generation script

`lifeforms/scripts/generate_species_md.py` — Claude Batch API runner.

Flow:
1. Query `world.db` `species` table for requested campaign roles
2. For each row with a `body_id`, call `meridian_api.get_body()` to fetch
   homeworld context
3. Serialise each row as YAML and submit as a Batch API request
4. Poll until the batch ends; write `.md` files to the vault

Model selection by role:

| Role | Model | Reason |
|---|---|---|
| PEOPLE | claude-opus-4-7 | Sapients are few, high-value, need coherence |
| BEAST | claude-sonnet-4-6 | Good prose, 3× lower cost |
| THING / MONSTER | claude-haiku-4-5-20251001 | Short entries, cost-sensitive |

System prompt is prompt-cached (ephemeral); ~1500 tokens, stable across all
species in a batch.

## 5. Unit conventions

GURPS source material uses US customary units throughout. All values stored in
`world.db` and passed through the pipeline use SI or IAU-standard units.

### Length

| Stored as | Unit | GURPS source | Conversion |
|---|---|---|---|
| `size_m` | metre (m) | yards | 1 yd = 1 m |

### Mass

| Stored as | Unit | GURPS source | Conversion |
|---|---|---|---|
| `mass_kg` | kilogram (kg) | pounds | 1 lb = 0.5 kg |

GURPS values are abstract game quantities, not physical measurements. Treat
them as already written in sensible units. A 200 lb creature is 100 kg; a
2-yard creature is 2 m. Do not carry sub-unit precision (centimetres, grams).

### Temperature

Use **kelvin** for planetary and astronomical contexts (mean surface temperature,
stellar data). Use **Celsius** when the context is human-scale: habitability
descriptions, biology, weather, comfort ranges.

Round to the nearest even degree Celsius unless precision matters. Precision
matters for physical constants used in calculations — melting points, boiling
points, phase boundaries. It does not matter for descriptive ranges ("cold",
"temperate", "hot").

### Pressure

Bar (1 bar = 100 kPa). Standard in planetary science; GURPS uses atmospheres
(atm) in descriptive text. 1 atm = 1 bar.

### Gravity

Standard gravities (g), where 1 g = 9.807 m s⁻². GURPS already uses this
unit; no conversion required. The `gravity_g` field stores the multiplier
directly (e.g. 0.40 for Mars-like gravity).

### Astronomical distances

| Scale | Unit |
|---|---|
| Within a star system | AU (astronomical unit; 1 AU = 1.496 × 10¹¹ m) |
| Stellar distances | light-year (ly) or parsec (pc); 1 pc = 3.086 × 10¹⁶ m |

GURPS quotes AU for orbital radii. No conversion needed; store as-is.

### Delta-V (GURPS Spaceships)

GURPS Spaceships quotes delta-V in **mps (miles per second)**. Convert as
**1 mps = 1.6 km/s**. Do not treat mps as km/s — that 38% error produces
significantly wrong travel times. Thrust in G requires no conversion.

### Damage Resistance (DR)

Dimensionless GURPS game stat. Stored as-is; no SI equivalent.
